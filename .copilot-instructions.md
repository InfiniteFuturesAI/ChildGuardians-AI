## 🧱 **Repository Policy**

### **Security Requirements**

**FORBIDDEN ACTIONS:**
- ❌ Include real law enforcement case numbers or references
- ❌ Include real officer names, badge numbers, or agency identifiers
- ❌ Include real victim identifiers or case details
- ❌ Include real CSAM hash values (even known ones)
- ❌ Expose production credentials, API keys, or tokens
- ❌ Include PII (names, emails, addresses, phone numbers)
- ❌ Bypass pre-commit hooks or secret scanning

**REQUIRED ACTIONS:**
- ✅ Use synthetic/demo data for all examples (label "DEMO ONLY")
- ✅ Replace real emails with `name@example.com`
- ✅ Replace hostnames with `example.com` or `agency.example.gov`
- ✅ Replace all keys/tokens with `DEMO_ONLY_XXXXXXXX`
- ✅ Use obviously fake case numbers (e.g., `CYBER-2024-00042`)
- ✅ Use obviously fake badge numbers (e.g., `SA-12345`)

## 🧠 **AI Agent Behavior Guidelines**

### **When Generating Examples**
1. **Always use synthetic data** - Never reference real cases or victims
2. **Label demo credentials clearly** - "DEMO ONLY - Do not use in production"
3. **Use realistic but fake patterns** - Real structure, fake content
4. **Document what's synthetic** - Make it obvious to reviewers

### **When Working with Evidence Systems**
1. **Never simulate real cases** - Only synthetic scenarios
2. **Never generate fake CSAM hashes** - Only hash synthetic test data
3. **Never include victim information** - Even synthesized victim data is inappropriate
4. **Document all security boundaries** - Explain what the system won't do

### **When Asked About Sensitive Topics**
1. **Redirect to documentation** - Point to SURVEILLANCE_EXCLUSION.md, SCOPE.md
2. **Emphasize system boundaries** - What this system is NOT
3. **Maintain professional tone** - This is law enforcement infrastructure
4. **Request escalation for edge cases** - When uncertain, ask

## �️ **Code Quality Standards (Lessons Learned)**

### **Before Committing ANY Code**

**AI Agents MUST ensure:**
1. **No trailing whitespace** - Ruff W293 errors break CI
2. **Proper exception chaining** - Use `raise X from err` not bare `raise X`
3. **Black formatting** - 100 char line length, run before commit
4. **Type hints** - Add return types and parameter annotations

**Prevention Commands (Run Before Commit):**
```bash
# Fix and format automatically
python -m ruff check --fix src/ tests/
python -m black src/ tests/ examples/

# Verify clean
python -m pytest -q
```

### **Common Errors We've Fixed**

| Error | Cause | Prevention |
|-------|-------|------------|
| W293 | Trailing whitespace on blank lines | Use Black formatter |
| B904 | Exception without `from` clause | Always use `raise X from err` |
| ARG002 | Unused function argument | Add `# noqa: ARG002` if intentional |
| Line too long | >100 chars | Black auto-wraps |
| Missing type hints | No annotations | Add `: type` and `-> ReturnType` |

### **File Hygiene Rules**

- **End files with newline** - Unix standard
- **Use 4 spaces** - Never tabs
- **UTF-8 encoding** - Always
- **No trailing whitespace** - On ANY line, including blank lines

## 🔲 **Failure Capture Doctrine**

> **If a system cannot produce an auditable failure artifact, it is not allowed to operate at speed.**

### **Required for CHILD GUARDIANS:**
- All evidence validation failures must be logged
- All chain of custody breaks must be documented
- All defense simulation failures must preserve context
- All verification failures must be reconstructible

### **Test Failures Are Valuable:**
- ❌ **FAILED TEST** markers indicate learning opportunities
- Preserve failure artifacts for analysis
- Document what was believed at moment of failure
- Enable reconstruction of failure context
